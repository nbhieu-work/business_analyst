# Extraction

## A. Business Rules
- **BR-01 (Consent):** The app must not record any GPS data unless the user has explicitly granted "Always" or "While Using" location permissions AND accepted the privacy policy prompt.
- **BR-02 (Local Buffering):** If the network is unavailable, GPS points must be stored locally in an SQLite database on the mobile device.
- **BR-03 (Batch Sync):** The app must upload GPS data in batches (e.g., up to 500 points per payload) rather than making an HTTP request for every single coordinate.
- **BR-04 (Deduplication):** The backend must silently ignore ingestion of any GPS coordinate that shares the exact same `uuid` generated by the client to prevent duplicates during retry scenarios.
- **BR-05 (Sampling Rate):** The SDK will only record a new point if the device has moved > 10 meters OR if > 60 seconds have passed since the last recorded point while "On Duty".

## B. Edge Cases
- **EC-01 (Low Power Mode):** If the device enters Low Power Mode, the background collection frequency should dynamically reduce (e.g., > 50 meters or > 5 minutes).
- **EC-02 (Storage Limit):** If the local SQLite DB exceeds 50,000 pending rows (e.g., device offline for weeks), the oldest rows should be overwritten (FIFO) to prevent local storage bloat.
- **EC-03 (Permission Revoked mid-session):** If the user revokes location permission from OS settings while the app is backgrounded, the app must gracefully halt collection and flush existing buffered data on the next foreground open.

## D. State Model
- **Mobile Point State (Local SQLite):**
  - `PENDING` -> Point created, ready for upload.
  - `SYNCING` -> Point is currently in HTTP transit.
  - `SYNCED` -> Server returned 202 Accepted (Point is physically deleted from SQLite shortly after).
- **Sync Worker State:**
  - `SLEEPING` -> Waiting for criteria (Wifi, Time elapsed).
  - `FLUSHING` -> Reading `PENDING` points and sending POST to server.

## E. Permissions Matrix
- **End User:** Can GRANT/REVOKE location tracking. Can trigger manual sync via app UI.
- **Admin/Operator:** (Not applicable for client-side extraction, applied on Server-side reporting).

## F. Non-functional Requirements
- **NFR-01 (Battery):** Background GPS service should consume < 3% battery per hour on an average device.
- **NFR-02 (Data Usage):** JSON payloads must be gzipped in transit to minimize cellular data usage.
- **NFR-03 (Security):** Data must be transmitted over TLS 1.2+ minimum. API must rate-limit by `user_id`.

## G. Traceability
- **BR-02, BR-03, NFR-01** -> Intake Constraints (Offline behavior, Battery bounds, Rate limits).
- **BR-04, Data Elements (uuid)** -> Intake Success Metrics (0% duplicates).
- **EC-02** -> Intake Constraints (Storage bounding).
